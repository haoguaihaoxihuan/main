1. 多线程写入 records 为什么要加 Lock？不加锁会出现什么问题？
因为多个线程同时执行 records.append() 属于 竞争条件。加锁（threading.Lock）保证每次只有一个线程执行追加操作，维护数据一致性。
不加锁会导致：
①数据错乱：列表内部结构可能被破坏，导致记录丢失、重复或程序崩溃。
②覆盖丢失：两条记录可能写入同一个内存位置，造成一条记录被另一条覆盖。
③不可预测：每次运行结果可能不同，调试极其困难。

2. 本案例多线程能提速，纯 CPU 计算场景还能加速吗？为什么？
本案例（IO 密集型）能提速。因为 time.sleep() 模拟网络/磁盘等待，线程在等待时释放 GIL，其他线程可执行，实现并发。
纯 CPU 计算（CPU 密集型）不能加速，甚至会变慢。

原因：Python 的 GIL（全局解释器锁） 限制同一时刻只有一个线程执行 Python 字节码。
多线程在 CPU 密集型任务中会频繁争抢 GIL，导致上下文切换开销，反而降低性能。
此时应使用 多进程（multiprocessing）或 异步 IO（asyncio）实现真并行。

3. 父类抛出 NotImplementedError 是什么设计思想，作用是什么？
设计模式：这是 “模板方法模式” 与 “接口约束” 的体现。
强制子类实现 —— 父类定义抽象接口，子类必须重写 predict 方法，否则调用时立即报错，避免“空实现”导致的静默错误。
约定优于配置 —— 明确告诉开发者“此方法应由子类提供具体逻辑”，提高代码可维护性。
运行时检查 —— 相比静态类型语言（如 Java 的 abstract），Python 采用这种动态方式，更灵活且符合 “鸭子类型” 哲学。
作用：作为“占位符”，确保所有子类都实现核心推理逻辑，避免因遗忘导致运行时返回无意义结果。
